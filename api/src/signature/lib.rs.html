11//! # Design
12//!
13//! This crate provides a common set of traits for signing and verifying
14//! digital signatures intended to be implemented by libraries which produce
15//! or contain implementations of digital signature algorithms, and used by
16//! libraries which want to produce or verify digital signatures while
17//! generically supporting any compatible backend.
18//!
19//! ## Goals
20//!
21//! The traits provided by this crate were designed with the following goals
22//! in mind:
23//!
24//! - Provide an easy-to-use, misuse resistant API optimized for consumers
25//!   (as opposed to implementers) of its traits.
26//! - Support common type-safe wrappers around "bag-of-bytes" representations
27//!   which can be directly parsed from or written to the "wire".
28//! - Expose a trait/object-safe API where signers/verifiers spanning multiple
29//!   homogeneous provider implementations can be seamlessly leveraged together
30//!   in the same logical "keyring" so long as they operate on the same
31//!   underlying signature type.
32//! - Allow one provider type to potentially implement support (including
33//!   being generic over) several signature types.
34//! - Keep signature algorithm customizations / "knobs" out-of-band from the
35//!   signing/verification APIs, ideally pushing such concerns into the type
36//!   system so that algorithm mismatches are caught as type errors.
37//! - Opaque error type which minimizes information leaked from cryptographic
38//!   failures, as "rich" error types in these scenarios are often a source
39//!   of sidechannel information for attackers (e.g. [BB'06])
40//!
41//! [BB'06]: https://en.wikipedia.org/wiki/Daniel_Bleichenbacher
42//!
43//! ## Implementation
44//!
45//! To accomplish the above goals, the [`Signer`] and [`Verifier`] traits
46//! provided by this are generic over a [`Signature`] return value, and use
47//! generic parameters rather than associated types. Notably, they use such
48//! a parameter for the return value, allowing it to be inferred by the type
49//! checker based on the desired signature type.
50//!
51//! The [`Signature`] trait is bounded on `AsRef<[u8]>`, enforcing that
52//! signature types are thin wrappers around a "bag-of-bytes"
53//! serialization. Inspiration for this approach comes from the Ed25519
54//! signature system, which was based on the observation that past
55//! systems were not prescriptive about how signatures should be represented
56//! on-the-wire, and that lead to a proliferation of different wire formats
57//! and confusion about which ones should be used. This crate aims to provide
58//! similar simplicity by minimizing the number of steps involved to obtain
59//! a serializable signature.
60//!
61//! ## Alternatives considered
62//!
63//! This crate is based on over two years of exploration of how to encapsulate
64//! digital signature systems in the most flexible, developer-friendly way.
65//! During that time many design alternatives were explored, tradeoffs
66//! compared, and ultimately the provided API was selected.
67//!
68//! The tradeoffs made in this API have all been to improve simplicity,
69//! ergonomics, type safety, and flexibility for *consumers* of the traits.
70//! At times, this has come at a cost to implementers. Below are some concerns
71//! we are cognizant of which were considered in the design of the API:
72//!
73//! - "Bag-of-bytes" serialization precludes signature providers from using
74//!   their own internal representation of a signature, which can be helpful
75//!   for many reasons (e.g. advanced signature system features like batch
76//!   verification). Alternatively each provider could define its own signature
77//!   type, using a marker trait to identify the particular signature algorithm,
78//!   have `From` impls for converting to/from `[u8; N]`, and a marker trait
79//!   for identifying a specific signature algorithm.
80//! - Associated types, rather than generic parameters of traits, could allow
81//!   more customization of the types used by a particular signature system,
82//!   e.g. using custom error types.
83//!
84//! It may still make sense to continue to explore the above tradeoffs, but
85//! with a *new* set of traits which are intended to be implementor-friendly,
86//! rather than consumer friendly. The existing [`Signer`] and [`Verifier`]
87//! traits could have blanket impls for the "provider-friendly" traits.
88//! However, as noted above this is a design space easily explored after
89//! stabilizing the consumer-oriented traits, and thus we consider these
90//! more important.
91//!
92//! That said, below are some caveats of trying to design such traits, and
93//! why we haven't actively pursued them:
94//!
95//! - Generics in the return position are already used to select which trait
96//!   impl to use, i.e. for a particular signature algorithm/system. Avoiding
97//!   a unified, concrete signature type adds another dimension to complexity
98//!   and compiler errors, and in our experience makes them unsuitable for this
99//!   sort of API. We believe such an API is the natural one for signature
100//!   systems, reflecting the natural way they are written absent a trait.
101//! - Associated types preclude multiple (or generic) implementations of the
102//!   same trait. These parameters are common in signature systems, notably
103//!   ones which support different digest algorithms.
104//! - Digital signatures are almost always larger than the present 32-entry
105//!   trait impl limitation on array types, which complicates trait signatures
106//!   for these types (particularly things like `From` or `Borrow` bounds).
107//!   This may be more interesting to explore after const generics.
108//!
109//! ## Unstable features
110//!
111//! Despite being post-1.0, this crate includes a number of off-by-default
112//! unstable features named `*-preview`, each of which depends on a pre-1.0
113//! crate.
114//!
115//! These features are considered exempt from SemVer. See the
116//! [SemVer policy](#semver-policy) above for more information.
117//!
118//! The following unstable features are presently supported:
119//!
120//! - `derive-preview`: for implementers of signature systems using
121//!   [`DigestSigner`] and [`DigestVerifier`], the `derive-preview` feature
122//!   can be used to derive [`Signer`] and [`Verifier`] traits which prehash
123//!   the input message using the [`PrehashSignature::Digest`] algorithm for
124//!   a given [`Signature`] type. When the `derive-preview` feature is enabled
125//!   import the proc macros with `use signature::{Signer, Verifier}` and then
126//!   add a `derive(Signer)` or `derive(Verifier)` attribute to the given
127//!   digest signer/verifier type. Enabling this feature also enables `digest`
128//!   support (see immediately below).
129//! - `digest-preview`: enables the [`DigestSigner`] and [`DigestVerifier`]
130//!   traits which are based on the [`Digest`] trait from the [`digest`] crate.
131//!   These traits are used for representing signature systems based on the
132//!   [Fiat-Shamir heuristic] which compute a random challenge value to sign
133//!   by computing a cryptographically secure digest of the input message.
134//! - `rand-preview`: enables the [`RandomizedSigner`] trait for signature
135//!   systems which rely on a cryptographically secure random number generator
136//!   for security.
137//!
138//! NOTE: the [`async-signature`] crate contains experimental `async` support
139//! for [`Signer`] and [`DigestSigner`].
140//!
141//! [`async-signature`]: https://docs.rs/async-signature
142//! [`digest`]: https://docs.rs/digest/
143//! [`Digest`]: https://docs.rs/digest/latest/digest/trait.Digest.html
144//! [Fiat-Shamir heuristic]: https://en.wikipedia.org/wiki/Fiat%E2%80%93Shamir_heuristic
